  • Each panel can have its own alt tag, making your comic accessible as well as getting better SEO. While this might not be perfect as a transcript, you can use aria-describedby to point to the transcript elsewhere on the page, and a screen reader user will get the same effect.
  • By designing a mobile panel layout and a desktop panel layout, you can make the panels reflow depending on how wide the screen is. No need to send people to WebToons or annoy desktop users with massive images as wide as the sun!
  • If you’re the type to use animated GIFs every now and then, you only need to animate one panel instead of the whole page. Easier on you and faster loading for the reader.
  • You now have the option of making individual panels interactive using JavaScript. Useless? Debatable. Cool? Definitely.
  • You can style each panel individually using CSS. Different borders for different panels? Hover states? CSS animations? It’s all there.

I can think of two potential downsides. First, it might be extra work for the creator to break your file into different files for each panel. This is true, but if you’re refactoring your comic for social media and meta cards anyway, this shouldn’t be much extra.

Secondly, as a performance thing, the browser is making a lot more calls to the server. Once for each file. The difference shouldn’t be noticeable except on slow networks, but the page should still work fine regardless. If you really want to side-step this you can do some extra work and save each panel as an SVG symbol in a single file. then each <img> tag will look for a xlink:use symbol instead of another server call. There’s also the easier option of putting your images on a CDN to begin with, but hey. You do you.

And now, an example, featuring “Pepper & Carrot” by David Revoy. Disclaimer: This example is not using reflowing, or alt tags. It is simply the comic as produced, replicated using CSS Grid.
